1) FINDINGS (INCONSISTENCIAS, ERROS E BUGS)
==================================================

1. Critico: utilizador suspenso pode continuar autenticado se ja tiver sessao ativa.

- apps/common/decorators.py:19

- Nao validam is_active/account_status em cada pedido.

2. Critico: convite admin pode ser reutilizado por links antigos para redefinir password/perfil.
Refs:
- apps/accounts/services.py:137
- apps/accounts/services.py:259
- apps/accounts/views.py:132
- apps/accounts/services.py:275
Motivo:
- Apenas o token usado no momento e marcado com used_at.

3. Alto: registo nao e transacional (risco de User orfao sem ProducerProfile).
Refs:
- apps/accounts/services.py:100
- apps/accounts/services.py:112
Motivo:
- Se falhar a criacao de ProducerProfile, o User pode ficar criado.
- Isso pode bloquear novo registo com o mesmo email.

4. Alto: criacao de utilizador por admin pode gerar erro 500 em email duplicado.
Refs:
- apps/dashboard/forms.py:24
- apps/dashboard/views.py:936
- apps/accounts/models.py:29
Motivo:
- Falta validacao de unicidade no fluxo admin + tratamento de IntegrityError.

5. Medio: endpoint de pedido de reset password sem rate limit.
Refs:
- apps/accounts/views.py:154
Comparacao:
- apps/accounts/views.py:32
- apps/accounts/views.py:73
Risco:
- Abuso/spam de emails de recuperacao.

6. Medio: login manual sem rotacao de session key (session fixation risk).
Refs:
- apps/accounts/services.py:320
Motivo:
- Nao existe renovacao explicita da chave de sessao no login.

7. Medio (operacional): envio async por thread daemon pode perder emails.
Refs:
- apps/accounts/services.py:62

- Em Railway (restart/deploy/scale), a thread pode ser interrompida.

8. Baixo (UX): template de convite admin nao mostra erros de password.
Refs:
- apps/accounts/forms.py:229
- templates/accounts/admin_invite_complete.html:55

9. Baixo: cobertura de testes praticamente inexistente em accounts.

- apps/accounts/tests.py:1

==================================================
2) FLUXO ATUAL (COMO ESTA A FUNCIONAR)
==================================================

1. Sessao/auth custom
- Middleware le session["user_id"] e define request.current_user.
- O projeto usa request.current_user como referencia principal de autenticacao.

2. Login ("/" e "/login/")
- Valida form.
- Aplica ratelimit no POST.
- Autentica por email/password com condicoes:
  - is_active = True
  - account_status = ACTIVE
- Grava sessao:
  - user_id
  - user_email
  - user_role
  - user_name
- Define expiracao por "remember me".

3. Registo publico ("/registo/")
- Cria User inativo.
- Cria ProducerProfile.
- Cria token de verificacao (24h).
- Envia email de confirmacao.
- Guarda registration_email na sessao.
- Redireciona para pagina de sucesso.

4. Verificacao de email ("/verificar-email/<token>/")
- Se token invalido/expirado: erro e redirect para login.
- Se token for ADMIN_INVITE: redirect para completar convite.
- Se token for SIGNUP_CONFIRMATION:
  - marca token usado
  - ativa conta (email_verified_at, is_active=True, account_status=ACTIVE)
  - sucesso e redirect para login.

5. Convite admin
- Admin cria utilizador pendente no dashboard.
- Sistema gera token ADMIN_INVITE (48h) e envia email.
- Utilizador abre link, completa perfil/password, conta fica ativa.

6. Recuperacao de password
- Pedido em "/recuperar-password/":
  - resposta sempre generica
  - se email ativo existir, gera token (2h) e envia email.
- Confirmacao em "/recuperar-password/<token>/":
  - valida token
  - altera password hash
  - marca token usado
  - redirect login.

7. Logout
- session.flush()

==================================================
4) RECOMENDACOES PRIORIZADAS
==================================================

1. Fechar imediatamente os pontos Criticos e Altos
- Validar is_active/account_status nos decorators/login_required e admin/client guards.
- Invalidar TODOS os tokens de convite ativos quando um convite e concluido.
- Tornar create_user_and_profile transacional (atomic).
- Adicionar validacao de email unico + tratamento de IntegrityError no create user admin.

2. Reforcar seguranca do fluxo de autenticacao
- Adicionar ratelimit no password_reset_request_view.
- Rodar/ciclar session key no login para mitigar session fixation.

3. Melhorar robustez de envio de emails em producao
- Substituir thread daemon por fila/job worker (ex: Celery/RQ/servico dedicado).
- Garantir retries e log estruturado de falhas.

4. Corrigir UX e observabilidade
- Mostrar erros de password no template admin_invite_complete.
- Limpar session["registration_email"] depois da pagina de sucesso.

5. Cobertura de testes minima recomendada
- Testes de auth/session para:
  - login ativo vs suspenso
  - registo + rollback transacional
  - convite admin com invalidez de tokens antigos
  - reset password com expiracao/uso unico
  - rate limits essenciais.

==================================================
5) RESUMO EXECUTIVO
==================================================

O fluxo base de login/registo/confirmacao/reset esta funcional, mas existem riscos
de seguranca e consistencia que devem ser tratados com prioridade, sobretudo:
- sessao de utilizador suspenso,
- reutilizacao de convites antigos,
- falta de atomicidade no registo,
- falta de controlo robusto no create user admin.
